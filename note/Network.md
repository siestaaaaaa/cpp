# 网络模型

## OSI模型

应用层：给应用程序提供统一的接口

表示层：把数据转换成另一个系统能识别的格式

会话层：管理表示层实体之间的会话

传输层：端到端的通信

网络层：数据的分片、路由、转发

数据链路层：数据的封帧、 MAC 寻址、差错检测

物理层：数据在物理网络中的传输

OSI 模型比较复杂，实际使用的一般是 TCP/IP 模型

## TCP/IP模型

### 应用层

应用层负责给应用程序提供统一的接口，应用层协议有 HTTP、HTTPS、DNS、FTP 等

应用层工作在用户态，其他层工作在内核态

### 传输层

传输层负责端到端的通信，传输层协议有 TCP、UDP 等

TCP 是可靠的、面向连接的、基于字节流的，UDP 是不可靠的、无连接的、基于数据报的

### 网络层

网络层负责数据的分片、路由、转发，网络层协议有 IP、ICMP 等

网络号和主机号：IP 地址和子网掩码进行按位与就能得到网络号，剩下的就是主机号

### 网络接口层

网络接口层负责数据在物理网络中的传输，网络接口层协议有 ARP、RARP 等

网络接口层工作在网卡这个层次，使用 MAC 地址来标识网络上的设备

## 为什么要有网络协议

不同设备的进程间通信，需要通过网络，为了兼容不同的设备，就需要一套通用的网络协议

协议就是一种约定

## 报文的封装和分用

不同协议层传输的数据都可以叫报文

对于每一层协议的报文，都可以看成报头 + 数据

发送端发送消息时，自顶向下的每层协议会给报文添加该层的报头

接收端接收消息时，自底向上的每层协议不断对报文进行解包分用

# 应用层

## HTTP

HTTP（超文本传输协议）是一个无连接、无状态的协议

### 请求报文

请求行：请求方法 URL 协议版本 \r\n

请求头：请求的附加信息，形式：`key: value\r\n`

空行：\r\n

请求体：请求的数据，通常用于 POST 请求等需要传输数据的情况

### 响应报文

状态行：协议版本 状态码 状态信息 \r\n

响应头：响应的附加信息，形式：`key: value\r\n`

空行：\r\n

响应体：响应的数据，通常是服务器返回的 HTML、JSON 等内容

### 常见状态码

1xx：信息性状态码，表示接收的请求正在处理

2xx：成功状态码，表示请求正常处理完毕

3xx：重定向状态码，表示需要进行附加操作以完成请求

4xx：客户端错误状态码，表示服务器无法处理请求

5xx：服务器错误状态码，表示服务器处理请求出错

### 常见请求方法

GET：获取指定资源

POST：提交数据

PUT：更新指定资源

DELETE：删除指定资源

HEAD：类似 GET，但只返回报头，不返回请求的资源

### GET和POST的区别

根据 RFC 规范：

GET 是安全且幂等的，安全是指不修改服务器上的数据，幂等是指多次请求和一次请求产生的影响相同，通常会对 GET 请求做缓存

POST 是提交数据的操作，会修改服务器上的资源，所以是不安全的，多次提交数据就会创建多个资源，所以是不幂等的，一般不会缓存 POST 请求

GET 通过 URL 传参，实际上是有长度限制的，POST 通过请求体传参，没有长度限制

### HTTP什么时候会断连

1. 当服务端或客户端执行 close 系统调用时，会发送 FIN 报文，然后四次挥手断开 TCP 连接
2. 当发送方发送了数据，接收方一段时间内没回复 ACK，发送方会重传，当重传达到最大次数时，就会断开连接
3. 连接上没有请求和响应，超过一定时间，就会自动断开连接

### 长连接

短连接：建立 TCP 连接 - 请求资源 - 响应资源 - 释放连接，一次连接只能请求一次资源

长连接：只要任意一端没有提出断开连接，则保持 TCP 连接，避免建立和释放连接的开销

HTTP/1.0 默认使用短连接，如果要使用长连接，需要在报头显式设置 `Connection: keep-alive`

HTTP/1.1 默认使用长连接，如果要使用短连接，需要在报头显式设置 `Connection: close`

### HTTP1.1怎么对请求做拆包

客户端在请求头添加 Content-Length 字段，指明请求体的长度，服务器拿到请求报文后，通过 \r\n 对请求行和请求头进行拆包，然后根据 Content-Length 的值来解析请求体

### HTTP2.0

1. 压缩头部
2. 二进制格式
3. 多个流复用一条连接
4. 允许服务器主动发送消息

### HTTP到底是不是无状态的

HTTP 本身是无状态的，因为每个 HTTP 请求都是独立的，服务器不会记录之前请求的状态，但是可以通过一些机制来保持状态，比如 Cookie 和 Session

### Cookie和Session

Cookie 和 Session 都是服务器用来跟踪用户状态的机制

- 用户首次访问网站时，服务器会将 Cookie 信息发送到客户端，客户端之后的请求时会携带 Cookie 信息，用于服务器识别用户、跟踪用户行为、记录用户偏好等
- 用户首次访问网站时，服务器会为用户创建一个唯一的 Session ID，通过 Cookie 或 URL 重写的方式发送到客户端，客户端之后的请求会携带这个 Session ID，服务器根据 Session ID 找到对应的 Session 信息

Cookie 和 Session 的区别：

- Cookie 存储在客户端，Session 存储在服务器
- Cookie 大小限制通常为 4KB，而且浏览器对每个域名的 Cookie 数量是有限制的，Session 理论上没有大小限制，主要受限于服务器内存大小
- Cookie 相对不安全，因为存储在客户端，容易被篡改或窃取，Session 相对更安全，因为存储在服务器，但也要防范 Session 劫持
- Cookie 信息随每个请求发送到服务器，可能影响网络传输效率，Session 信息存储在服务器上，每次请求通过 Session ID 查询服务器上的 Session 信息，可能增加服务器负载

### HTTP和HTTPS

HTTP 是明文传输，不安全，所以 HTTPS 在 HTTP 和 TCP 之间加入了 SSL/TLS 来加密数据传输

HTTP 在 TCP 三次握手建立连接之后就可以进行传输，HTTPS 还要进行 SSL/TLS 的握手过程：

1. 客户端发起请求（包含支持的加密算法）
2. 服务器选择加密算法，发送数字证书（内含公钥）
3. 客户端验证证书，生成预主密钥，用服务器给的公钥加密发给服务器
4. 服务器用私钥解密得到预主密钥
5. 双方通过预主密钥生成会话密钥

SSL/TLS 的握手过程是非对称加密（客户端用服务器给的公钥加密消息，服务器用私钥解密消息），后续通信是对称加密（使用协商好的加密算法和会话密钥进行加密）

HTTP 默认端口 80，HTTPS 默认端口 443

### HTTP和RPC

RPC 是一种调用方式，期望像调用本地方法那样去调用远端的服务方法（RPC 有很多实现方式，不一定基于 TCP）

HTTP 主要用于 B/S 架构，RPC 更多用于 C/S 架构。很多软件同时支持多端，一般对外 HTTP，内部用 RPC

RPC 出现的比 HTTP 早，且比主流的 HTTP/1.1 性能更好，所以大部分公司内部还在使用 RPC

### HTTP长连接和WebSocket

HTTP 是半双工的，WebSocket 是全双工的

实时性要求较高的场景，可以使用 WebSocket

## DNS

DNS 是用于将域名转换为 IP 地址的分布式系统

DNS 是基于 UDP 的，实时性较好，但是有丢包的风险，所以 DNS 内部使用了一些机制（比如超时重传）来提高可靠性

DNS 默认端口是 53

# 传输层

传输层负责端到端的通信

TCP 和 UDP 都是全双工的

## TCP

TCP 是可靠的、面向连接的、基于字节流的

### TCP报头

源端口，目的端口：端到端

序列号：初始序列号随机生成，之后每个字节的数据都会使序列号加 1，用来解决网络包乱序的问题

确认号：确认号之前的数据被认为正常接收，用来解决丢包的问题

报头长度，标志位，窗口大小

校验和，紧急指针

选项

### 为什么要三次握手

防止历史连接建立而造成资源浪费

考虑网络拥堵的情况：旧 SYN 比新 SYN 先到达服务端，服务端对旧 SYN 回复 ACK，客户端收到 ACK 发现确认号对不上，就会返回 RST，服务端收到 RST 就会释放历史连接，由最新的 SYN 来建立连接。如果是两次握手，服务端就没有一个中间状态来阻止历史连接

### 四次握手可以吗

可以，但是 ACK+SYN 一般可以优化成一步

### 中间两次挥手可以合并吗

取决于被动关闭方是否还有数据要发送

如果有，则必须由用户程序发送完数据后，再发起第三次挥手

如果没有，中间两次挥手可以合并

### 超时重传机制

数据包丢失（发送方未收到 ACK）或者确认包丢失（接收方收到重复数据），都会触发超时重传。重传的报文序列号不变

最大重传次数一般受系统参数控制，每次超时时间是上一次的两倍。达到最大重传次数后，会自动断开连接

### 大量发送SYN包会怎么样

有可能导致接收端的 TCP 半连接队列满了，后面收到的 SYN 就会丢包，导致无法建立连接

如何解决 SYN 攻击：

- 调大 netdev_max_backlog（接收数据包的速度大于内核处理的速度时，会有一个队列保存这些数据包）
- 增大 TCP 半连接队列
  - 增大 net.ipv4.tcp_max_syn_backlog
  - 增大 listen() 函数中的 backlog
  - 增大 net.core.somaxconn
- 开启 net.ipv4.tcp_syncookies（绕过半连接队列建立连接）

### 粘包怎么解决

TCP 是基于字节流的，没有边界，会有粘包的问题

最简单的方式是固定消息长度，但是这种方式不够灵活

1. 可以使用特殊字符作为边界，比如 HTTP 的 \r\n，但是要注意数据中对应的特殊字符要转义
2. 还可以自定义消息结构，在消息头部存一个字段来说明紧随其后的数据大小

### 基于TCP的协议

HTTP、HTTPS、SSH、Telnet、FTP、SMTP

## UDP

UDP 是不可靠的、无连接的、基于数据报的

### UDP报头

源端口，目的端口：端到端

报文长度，检验和

### UDP特点

不可靠：

- 没有确认机制和重传机制
- 不会给应用层返回任何错误信息
- 接收缓冲区不能保证接收报文的顺序和发送报文一致
- 如果接收缓冲区满了，再到达的数据报会被丢弃

无连接：不需要建立连接

基于数据报：

- 不能控制一次读写多少数据
- 如果传输的数据很大，需要在应用层手动分包，接收端手动拼装

### 基于UDP的协议

DNS、DHCP、NFS

## TCP和UDP

TCP 是可靠的，UDP 不保证可靠

TCP 传输数据前需要建立连接，UDP 不需要

TCP 是基于字节流的，没有边界，保证顺序和可靠，UDP 是基于数据报的，有边界，可能丢包或乱序

TCP 用于需要可靠传输的场景，UDP 用于实时性要求较高的场景

## TCP为什么可靠

连接管理：TCP 需要三次握手建立连接、四次挥手释放连接

序列号：TCP 对每个字节的数据都进行了编号，防止数据丢失、重复、乱序

确认机制：接收方收到数据，会回复 ACK 报文给发送方，否则发送方将会超时重传

超时重传：数据包丢失，超时重传数据包。确认包丢失，超时重传确认包

流量控制：根据接收端接收能力控制发送速度（滑动窗口）

拥塞控制：根据网络拥堵情况控制发送速度（拥塞窗口）

## 滑动窗口

发送缓冲区中维护了一个滑动窗口，滑动窗口左侧是已确认的数据，右侧是未发送的数据，窗口内是可以直接发送的数据

滑动窗口的大小受 TCP 报头中的窗口大小字段控制

## 拥塞窗口

发送端

## UDP实现可靠传输

将 TCP 的特性在应用层实现即可

- 引入序列号，保证数据顺序
- 引入确认机制，确保收到数据
- 引入超时重传、流量控制、拥塞控制等

# 网络场景

## 从输入网址到网页显示

1. 浏览器解析 URL，生成请求
   - URL = 协议 + 服务器地址 + 端口（知名端口可以省略）+ 文件路径
   - 文件路径如果为空，则访问 web 根目录下的 index.html
2. 查询服务器域名对应的 IP 地址
   - 客户端先检查缓存，如果没有再向本地 DNS 服务器请求
   - 本地 DNS 服务器如果没有，会向根 DNS 服务器请求，直到获取到目标服务器的 IP 地址
3. 三次握手建立 TCP 连接
   - 客户端发送 SYN（在吗）
   - 服务端收到 SYN，内核会把连接存储到半连接队列，并回复 ACK+SYN（在，在吗）
   - 客户端收到 ACK+SYN，返回 ACK（在），这次握手可以携带数据
   - 服务端收到 ACK，内核会把连接从半连接队列移到全连接队列（accept 就是把连接从全连接队列中取出来给用户使用）
   - 三次握手保证了双方都有发送和接收的能力
4. 客户端发送请求
5. 网络传输
   - 网卡 - 交换机 - 路由器 - ... - 路由器 - 交换机 - 网卡
   - 源 IP 和 目标 IP 始终不变（前提是没有使用 NAT）
   - 源 MAC 和目标 MAC 会一直变化
6. 服务端接收请求
7. 服务端处理请求
8. 服务端生成响应
9. 服务端发送响应
10. 客户端接收响应
11. 客户端处理响应
12. 四次挥手断开 TCP 连接
    - 主动关闭方发送 FIN（我说完了）
    - 接收方收到 FIN，回复 ACK（好），继续发送没发完的数据
    - 数据发完了返回 FIN（我也说完了）
    - 主动关闭方收到 FIN，回复 ACK（好），然后等待 2MSL 时间（确保最后的 ACK 到达、使该连接上的所有报文都失效），连接彻底关闭

## 服务器ping不通但HTTP请求成功

ping 基于 ICMP 协议，HTTP 基于 TCP 协议，有可能是服务器的防火墙禁止了 ICMP 协议，没有禁止 TCP 协议
