### Shell

Shell 是一个应用程序，介于用户和内核之间，用户通过 Shell 与内核进行交互（Windows 的图形化界面就是 Shell）

Linux 常用的 Shell 是 Bash，可以使用命令行交互，也可以进行脚本编程

```bash
# 创建 bash 脚本
vim test.sh

# 执行 bash 脚本
bash test.sh
# 或者
chmod +x test.sh
./test.sh

# 定义变量  注意不能加空格  shell变量其实都是字符串
name="siestaaaaaa"

# 使用变量  可以加 {}
echo $name
echo ${name}

# 删除变量
unset name

# 特殊变量
echo $0 # 脚本名
echo $1 # 脚本参数 $2 $3 ...
echo $# # 脚本参数个数
echo $* # 脚本所有参数
echo $? # 上一条命令的退出码

# 打印字符串
echo "my name is ${name}!"

# 获取字符串长度
echo ${#name}

# 获取子串  :begin:n
echo ${name:1:4}

# 数组变量
arr=(0 1 2 3 4)
echo ${arr[0]}
arr[100]=100
echo ${arr[100]}

# 获取数组所有元素
echo ${arr[@]}

# 获取数组长度
echo ${#arr[@]}

# 获取数组元素长度
echo ${#arr[0]}

# 关联数组  可以用字符串当键
declare -A mp
mp["1"]=1
mp["2"]=2

# 获取关联数组所有值
echo ${mp[@]}

# 获取关联数组所有键
echo ${!mp[@]}
```

# 硬件结构

## 冯诺依曼体系结构

冯诺依曼体系结构遵循了图灵机的设计，约定用二进制进行计算和存储，将计算机分为五个部分：运算器、控制器、存储器、输入设备、输出设备

运算器和控制器是 CPU 的一部分

## 位宽

CPU 位宽：CPU 一次能计算的数据宽度。硬件的位宽指的就是 CPU 位宽

软件的位宽：程序指令的位宽。操作系统也是软件

思考：32 位的 CPU 可以运行 64 位的操作系统吗

## 程序的执行过程

执行程序就是 CPU 执行程序编译后的一条条指令

例子：a=1+2 的执行过程

假设 a=1+2 被编译成下面四条指令

1. 将 1 这个数据从内存加载到寄存器
2. 将 2 这个数据从内存加载到寄存器
3. 将两个寄存器的值相加，结果存到寄存器
4. 将结果从寄存器写到 a 这个变量的地址处

## 存储器的层次结构

CPU 寄存器、CPU Cache（SRAM）、内存（DRAM）、硬盘...

每层的存储器只和相邻层打交道

# 操作系统

## 操作系统是什么

操作系统是一款管理软硬件资源的软件

## 系统调用

系统调用就是操作系统向用户提供的 API，用于用户和内核的交互

调用系统调用，操作系统会从用户态切换到内核态

## 用户态和内核态

用户态和内核态是操作系统的两种运行模式

- 用户态的权限低，无法访问硬件资源，用于运行用户程序
- 内核态的权限高，可以访问硬件资源，用于运行 OS 内核

通过区分用户态和内核态，可以更好的控制权限，提高操作的安全性，并且用户态程序崩溃不会影响系统，可以提高系统的稳定性

## 中断

用户态到内核态的切换是通过中断完成的

中断发生时，当前进程会阻塞，然后操作系统切换到内核态对中断进行处理，不同的中断有不同的处理方式

# 进程管理

## 进程概念

进程就是正在运行的程序

PCB 是进程存在的唯一标识，管理着进程的各种信息（pid、uid、状态、优先级等）和资源（地址空间、页表、文件描述符表等）

## 进程状态

进程状态有：创建、就绪、运行、阻塞、挂起、结束

运行状态的进程，如果时间片用完，就会变成就绪状态

进程等待事件会进入阻塞状态，事件完成时变为就绪状态

阻塞的进程可能会被换出到硬盘，这个状态就是挂起

## 僵尸进程

僵尸进程已经结束了，但是 PCB 等资源没有释放

僵尸进程是怎么产生的：一个进程结束时，它的父进程存在、且没有调用 wait/waitpid 等待子进程结束、也没有忽略 SIGCHLD 信号

如何解决僵尸进程：

- 杀掉父进程，使僵尸进程变成孤儿进程。孤儿进程会被 1 号进程接管
- 父进程调用 wait/waitpid 等待子进程结束（可以捕捉 SIGCHLD 信号，在捕捉函数中调用 wait/waitpid）
- 父进程忽略 SIGCHLD 信号。由内核对僵尸进程进行回收

## 写时复制

进程通过 fork 创建子进程时，会把进程的资源复制一份给子进程，但是子进程的地址空间对应的物理内存不会复制，这样子进程就共享了父进程的物理内存数据，可以节约物理内存资源

当子进程对内存进行写操作时，操作系统才进行物理内存的复制，并重新设置页表的映射关系

## 进程程序替换

进程程序替换，替换的是代码段，不会替换进程资源，常用手法：fork + exec

exec 系列函数（`exec + l/v + 可选p e`）

- 函数参数：程序所在路径，命令行参数，环境变量（如有）
- `l`：传命令行参数列表
- `v`：传命令行参数数组
- `p`：路径会被 PATH 解析
- `e`：传环境变量数组
- 成功不返回，失败返回 -1 并设置 errno

## 进程线程协程

进程就是正在运行的程序，是系统资源分配的基本单位。进程之间的独立性高，每个进程都有自己的地址空间、页表、文件描述符表等资源，创建、销毁、切换的开销较大，一般用于执行多个独立的任务

线程是进程内的一个执行单元，是 CPU 调度的基本单位。线程共享进程的资源，只有自己的栈和寄存器，创建、销毁、切换的开销比进程小，适合计算密集的并发场景

> Linux的线程是用轻量级进程实现的，可以用 `ps -aL` 命令查看

协程是一种用户态的轻量级线程，协程的调度是协作式的，由程序自己来调度，可以很方便的写出异步非阻塞的代码。协程创建、销毁、切换的开销很小，但是不能很好的利用多核 CPU，适合 IO 密集的并发场景

## 并行和并发

并行：一个时刻有多个事件发生

并发：一段时间有多个事件发生

## 线程安全

多线程下，程序的执行结果是确定的，调用端无需做额外的同步操作

## 一个进程可以创建多少个线程

一个进程可以创建多少个线程，和几个东西有关：

- 虚拟内存的大小
- 系统参数（`/proc/sys/kernel/threads-max`、`/proc/sys/kernel/pid_max`）

## 调度算法

先来先服务：如果一个长作业先运行了，后面短作业的等待时间就会很长

短作业优先：会导致长作业的等待时间很长

高响应比优先：响应比 =（等待时间 + 服务时间）/ 服务时间，权衡了长作业和短作业。但是服务时间没法提前知道，无法实现

时间片轮转：最公平。时间片太短会导致频繁的上下文切换，太长会导致短作业的响应时间较长

最高优先级：可能导致低优先级的进程永远不会运行

多级反馈队列：新进程加入最高级队列，如果时间片用完了进程还没结束，就转入到下一级队列，队列级别越低，时间片越长。如果更高级队列中有新进程加入，就立刻停止当前运行的进程，转而去调度高级队列。这样兼顾了长作业和短作业，同时有较好的响应时间

## 死锁

死锁的四个必要条件：互斥条件、持有并等待条件、不可剥夺条件、循环等待条件

解决死锁：破坏死锁的任意一个必要条件即可，常见做法是以固定顺序获取锁来破坏循环等待条件

# 内存管理

## 地址空间

操作系统会给每个进程分配一套独立的虚拟地址空间，每个地址标识一个字节

进程地址空间包括用户空间和内核空间，每个进程都有自己的用户空间，内核空间是所有进程共享的

用户态只能访问用户空间，内核态才能访问内核空间

用户空间从低到高分为代码段、数据段、堆、栈

进程地址空间的作用：

- 保证每个进程的地址独立，将进程管理和内存管理解耦
- 多了一层虚拟地址，可以更好的控制访存安全
- 可以利用硬盘的空间，增加可用内存的大小

## 分段

分段是把进程地址空间划分为不同的逻辑段，每个段通过段表映射到物理内存中

由于每个段的大小不固定，所以可能有外碎片（内存碎片指的是物理内存）

解决方式：分页

## 分页

分页是把进程地址空间划分为固定大小的页面，每个页面通过页表映射到物理内存中

由于分页机制分配内存的最小单位是一页，每个页的大小是固定的，所以不会有外碎片，但是可能有内碎片

> Linux 的内存管理是把进程地址空间先分段，每个段再分页

## 多级页表

页面大小一般为 4KB，较大的页面大小可以减少表项数量，减少页表占用的空间，较小的页面大小可以减少内碎片，提供更细粒度的内存管理

为了减少内碎片，页面大小一般比较小，这样会导致页表占用空间较大。可以使用多级页表来解决，一级页表保存到二级页表的映射，这样需要保存的表项就比较少，同时二级页表可以按需创建。多级页表以此类推

多级页表也有缺点，就是虚拟地址和物理地址的转换多了几层，多了时间开销。可以用页表缓存保存最常访问的页

## MMU怎么工作的

当进程访问一个虚拟地址时，MMU 会将虚拟地址解析成页号和页内偏移量，然后查找页表，根据页号找到对应的页表项，得到物理页的地址，物理页的地址加上页内偏移量得到对应的物理地址

转换过程中可能还会涉及多级页表、页表缓存等机制

## 缺页中断

当进程访问的虚拟地址在页表中查不到时，系统会产生一个缺页中断，切换到内核态分配物理内存，更新页表，然后切回用户态，恢复进程的运行

## malloc怎么分配内存

小块内存调 brk()，从堆中分配内存，释放后回到内存池

大块内存调 mmap()，从文件映射区分配内存，释放后归还操作系统

阈值由实现定义，一般是 128KB

malloc 返回的地址前还有一块内存存储了内存的元信息，这样 free 的时候就知道要释放多大的内存

malloc 申请的是虚拟内存，并没有真正分配物理内存。当 CPU 访问这块虚拟内存的时候，产生缺页中断，内核才分配物理内存。如果有空闲的物理内存就直接分配，否则将进行内存回收。如果回收后还是没有足够大的空间，就会触发 OOM 机制，根据算法杀掉进程，直到有足够大的空间来分配

## 页面置换算法

内存回收是基于页面置换算法进行的，页面置换算法会选择一个物理页换出到硬盘，然后把需要访问的页换入到物理页

页面置换算法需要尽可能减少页面的换入换出次数

OPT：置换在未来最长时间不访问的页面，实际中无法实现

FIFO：置换在内存驻留时间最长的页面

LRU：置换最长时间没有被访问的页面

LFU：置换访问次数最少的页面

## 如何保证一个进程不被OOM杀掉

可以调整 oom_score_adj 为 -1000 来防止一个进程不被 OOM 杀掉

## 在4G机器上申请8G内存

32 位操作系统下，虚拟地址空间最大为 4G，申请 8G 虚拟内存会失败

64 位操作系统下，申请 8G 虚拟内存是没问题的，只要没有使用，操作系统就不会真正分配物理内存，如果使用了这块内存，开启 Swap 机制可能可以解决问题（Swap 机制会把不常用的内存换出到硬盘）

# 进程间通信

## 管道

管道分为匿名管道和命名管道，匿名管道只能用于有亲缘关系的进程之间的通信，命名管道可以用于任意进程之间的通信

管道的本质是通过内核缓冲区传输数据，命名管道存在于文件系统的作用是作为一个标识符，让进程能访问到同一块缓冲区

管道是半双工通信，传输的是字节流

## 消息队列

消息队列是保存在内核中的消息链表，可以传输自定义类型的数据

消息队列的通信不是最及时的，每次读写数据都要经过用户态和内核态之间的拷贝

## 共享内存

共享内存就是将同一块物理内存映射到多个进程的地址空间

共享内存是最快的进程间通信方式，没有用户态和内核态之间的数据拷贝

共享内存需要自己保证互斥和同步

## 信号

信号是一种异步通信机制

信号的处理方式有：捕捉、忽略、默认

捕捉信号就是提供一个处理函数，内核在处理信号时会切换到用户态执行这个函数

有两个信号是应用进程无法忽略和捕捉的：SIGKILL、SIGSTOP

## 信号量

信号量是一个计数器，通过两个原子操作 P、V 来控制，用于实现互斥和同步

信号量非负时表示可用资源的数量，为负时表示阻塞线程的数量

互斥信号量初始值为 1，同步信号量初始值为 0

## Socket

> Socket = IP 地址 + 端口，用于描述通信链路的一端

不同主机的进程间通信需要通过 Socket，Socket 也可以用于本地主机的进程间通信

Socket 常见的通信方式：基于 TCP 的通信、基于 UDP 的通信、本地进程间通信

> 客户端 Socket 的端口是随机的（OS 会自动 bind），服务端 Socket 的端口是固定的（需要手动 bind）

1. TCP 客户端：socket - connect - 数据传输

   TCP 服务器：socket - bind - listen - accept - 数据传输

2. UDP 客户端：socket - 数据传输

   UDP 服务器：socket - bind - 数据传输

## 线程间通信

同一进程内的线程共享进程的资源，只要是共享资源就可以做到线程间通信

线程间通信关注的是多线程同步的问题，常见的通信方式有：互斥锁、条件变量、自旋锁、读写锁、信号量

# 文件系统

## 一切皆文件

Linux 下一切皆文件，文件、目录、管道、套接字等都是交给文件系统管理的

索引结点（inode）：文件的唯一标识，记录文件的元信息（不含文件名），指向实际的数据块

每个文件系统都有自己的索引结点结构

## 硬链接

硬链接就是多个文件名指向同一个索引结点

删除文件实际上只是减少硬链接数，当文件的硬链接数为 0 时，操作系统才会真正删除这个文件

索引结点是不跨文件系统的，所以硬链接是无法跨文件系统的

## 软链接

软链接（符号链接）相当于一个新的文件，这个文件的内容是另一个文件的文件名

软链接是可以跨文件系统的

删除软链接或其指向的文件，互相都不会受影响

## 打开文件

进程打开一个文件时，内核会创建一个文件对象，并分配一个文件描述符指向这个文件对象

每个进程都有一个文件描述符表，文件描述符就是文件描述符表的下标

通过 fork() 创建一个子进程时，子进程会拷贝一份相同的文件描述符表，文件对象的引用计数加 1

关闭一个文件描述符时，文件对象的引用计数减 1，当文件对象的引用计数为 0 时，文件才真正关闭

## 重定向

重定向就是改变文件描述符的指向

系统调用：

```cpp
int dup(int fd); //把fd的指向拷贝到某个文件描述符并返回
int dup2(int fd, int fd2); //把fd的指向拷贝到fd2并关闭fd2原本指向的文件
```

bash 命令：

```bash
0  # 标准输入的文件描述符
1  # 标准输出的文件描述符
2  # 标准错误的文件描述符

command > file  # 把 1 重定向到 file
command >> file # 把 1 追加重定向到 file
command < file  # 把 0 重定向到 file

command 2> file # 把 2 重定向到 file
command &> file # 把 1 和 2 重定向到 file

command 1> file1 2> file2 # 把 1 重定向到 file1, 2 重定向到 file2
```

重定向和管道：重定向是改变进程某个文件描述符的指向，管道是两个进程间进行通信

## 缓冲区

标准输出：行缓冲

标准错误：无缓冲

普通文件：全缓冲

# 网络I/O

## I/O的过程

I/O 操作分为两个过程：准备数据、拷贝数据（用户态和内核态之间）

数据准备好了，就是 I/O 就绪

数据拷贝完了，就是 I/O 完成

## I/O模型

I/O 模型有五种：阻塞 I/O 模型、非阻塞 I/O 模型、I/O 复用模型、信号驱动 I/O 模型、异步 I/O 模型

前四种 I/O 模型都是同步 I/O，只有异步 I/O 模型是异步 I/O

同步 I/O：内核向应用程序通知的是 I/O 就绪，然后由应用程序来拷贝数据

异步 I/O：内核向应用程序通知的是 I/O 完成，整个 I/O 操作都由内核完成

### 阻塞I/O

阻塞 I/O 会阻塞等待直到 I/O 完成，套接字默认都是阻塞 I/O

阻塞 I/O 适用于并发量较小的场景

### 非阻塞I/O

非阻塞 I/O 不会阻塞等待准备数据的过程，而是直接返回

非阻塞 I/O 适用于实时性要求不高的场景

### I/O多路复用

I/O 多路复用可以同时等待多个 I/O 操作，当其中任何一个 I/O 操作就绪时，会通知应用程序拷贝数据

I/O 多路复用适用于需要同时处理多个 I/O 操作的场景，比如高并发服务器（Reactor 模式）

Linux 实现 I/O 多路复用的方式：select、poll、epoll

select 和 poll 都是将已连接的 socket 放到一个文件描述符集合，然后将这个集合拷贝到内核，由内核来轮询检查是否有事件产生。当事件产生时，select/poll 会标记对应的文件描述符，再将整个集合拷贝回用户态，用户通过遍历找到要处理的 socket

select 和 poll 的区别在于，select 使用固定长度的位图来表示文件描述符集合，支持的文件描述符个数是有限的。poll 使用的是链表，文件描述符个数没有限制（当然会受系统资源限制）

epoll 在内核里维护了一棵红黑树，来保存需要监听的 socket，不需要像 select/poll 那样每次都要把 socket 集合拷贝到内核。当事件产生时，epoll 会将对应的事件加入到就绪链表，用户不需要像 select/poll 那样遍历集合来找到要处理的 socket。另外，epoll 的文件描述符个数也没有限制（当然会受系统资源限制）

select 和 poll 只支持水平触发（不断通知直到数据读完），epoll 还可以设置边缘触发（只通知一次，一般和非阻塞 I/O 一起使用）

### 信号驱动I/O

信号驱动 I/O 就是应用程序发起 I/O 操作时，向内核注册一个信号处理函数，然后直接返回，当 I/O 就绪时，内核会通过信号通知应用程序，在信号处理函数中完成数据的拷贝

信号驱动 I/O 适用于需要异步通知的场景

### 异步I/O

异步 I/O 的整个 I/O 过程都由内核完成，应用程序不做任何等待，I/O 完成后，内核会通知应用程序

异步 I/O 需要操作系统的支持（Linux 的异步 I/O 是不完善的），适用于高性能高并发的场景（Proactor 模式）

## 高性能网络模式

下面介绍两种基于事件驱动的高性能网络模式

### Reactor模式

Reactor 模式是一种同步非阻塞的网络模式，通过 I/O 多路复用来监听事件，收到 I/O 就绪事件后，将事件分发给某个进程或线程，由应用程序完成数据的拷贝

Reactor 模式实现起来相对简单，对于耗时短的并发任务也足够高效；但是对于耗时长的并发任务会导致事件分发的阻塞，影响后续事件的处理

### Proactor模式

Proactor 模式是一种异步非阻塞的网络模式，应用程序只负责发起 I/O，整个 I/O 过程由内核完成并通知应用程序

Proactor 模式性能更好；但是实现更复杂，且依赖操作系统对异步的支持

Windows 的异步支持好，但是 Windows 系统不适合用于服务器；Linux 的异步支持不完善。所以主流的高并发服务器大多采用 Linux 下的 Reactor 模式

## 零拷贝

传统 I/O 方式存在较多的数据拷贝和上下文切换，消耗了大量的 CPU 资源

零拷贝就是尽量减少数据拷贝和上下文切换，常见的零拷贝技术有：

- DMA：允许外设直接访问内存，无需通过 CPU
- sendfile 系统调用：如果程序不关心文件的具体内容，可以使用 sendfile 将文件直接在内核态发送，无需拷贝到用户态
- 共享内存：最快的进程间通信方式，没有用户态和内核态之间的数据拷贝
- PageCache：缓存最近被访问的数据，当空间不足时淘汰最久未被访问的缓存。PageCache 还有预读和异步写入的功能，可以提高文件读写效率

大文件传输不应该使用 PageCache（而是直接用异步 I/O），因为 PageCache 可能会被大文件占据，热点小文件无法利用到 PageCache，并且大文件的缓存命中不高，还多了拷贝到 PageCache 的开销，可能导致性能下降
