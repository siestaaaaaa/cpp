# UObject

UObject 是 UE 对象系统的基类，具有完整的引用链，提供了通用的属性和接口、统一的 GC、反射、序列化

## GC

继承 UObject 且用 UPROPERTY 标记的成员变量，会参与 GC

UE 采用的 GC 算法是标记-清理

1. 获取 GC 锁，暂停其他线程对 UObject 引用关系的修改
2. 从 Root Set 开始，遍历所有可达对象并标记为存活 
3. 回收不可达对象
4. 释放 GC 锁

### 内存泄漏

有 GC 也会内存泄漏吗？

会！忘记删除引用，GC 就一直无法回收对象，从而内存泄漏

## 反射

反射就是运行时获取类的信息

## 序列化

序列化就是将内存中的数据转换成另一种格式，用于持久化或网络传输，反序列化反之

UE 的序列化只会序列化必要的数据，如果成员变量的值等于默认值，则不会序列化（CDO）

### 指针的序列化

由于每次反序列化，对象的地址会改变，所以指针的序列化不能存储地址

UE 用一张表来记录被指针指向的对象，序列化指针的时候，存储的是表的索引

# 联机开发

## ENetMode

NM_Standalone：单机模式，没有网络连接，不区分客户端和服务器

NM_Client：纯客户端，游戏只跑客户端逻辑

NM_DedicatedServer：专用服务器，游戏只跑服务器逻辑

NM_ListenServer：监听服务器，允许其他客户端连接，游戏既充当客户端又充当服务器

总结：

1. 只要不是 DS，都要跑客户端逻辑
2. 只要不是 Client，都要跑服务器逻辑

## Actor Role

游戏中的每个 Actor 都有自己的 Role

- ROLE_None：表示该 Actor 没有网络角色
- ROLE_SimulatedProxy：表示该 Actor 是一个模拟代理，通常用于客户端，客户端会接收来自服务器的状态更新，并在本地模拟该 Actor 的行为
- ROLE_AutonomousProxy：表示该 Actor 是一个自主代理，通常用于客户端，客户端可以直接控制该 Actor，并向服务器发送输入和状态更新
- ROLE_Authority：表示该 Actor 具有权威性，通常用于服务器，服务器负责管理该 Actor 的状态，并处理所有的游戏逻辑

## 属性同步

给属性标记 Replicated，并实现 GetLifetimeReplicatedProps 函数，当该属性在服务器上发生变化，就会自动同步到客户端

# 网络

## 延迟

延迟是不可避免的

- 从微观上来看，计算机通过中断获取键盘输入，这个过程也是存在延迟的
- 在游戏引擎中，逻辑线程将任务分发到渲染线程，渲染线程最快也要下一帧才开始处理
- 常用的 buffer 技术，一定程度上也会增大延迟

网络游戏中，最大的延迟一般是网络延迟，也就是 RTT

## 网络协议

游戏最常用的网络协议是 UDP，通常会实现一套可靠的 UDP，然后混用可靠的 UDP 和不可靠的 UDP

用户输入、逻辑事件等重要消息，一般使用可靠的 UDP

纯表现的消息或者状态同步，可以使用不可靠的 UDP

## 移动同步

客户端每帧上报自己模拟的移动结果和用户的移动输入，服务器收到后会去模拟角色的移动

如果服务器模拟的结果和客户端上报的差别不大，则会采纳客户端模拟的结果，否则使用服务器模拟的结果

最后，服务器将这帧所有角色的移动结果，同步到所有客户端

# 渲染

## 三种线程

游戏线程并非直接向渲染线程派发任务，而是不断将任务添加到一个队列，渲染线程会异步地从队列里取出任务进行处理

渲染线程也并非直接向 GPU 发送指令，而是不断将渲染命令添加到命令列表，由 RHI 线程异步地从命令列表中取出命令，向 GPU 发送

从而使得 CPU 无需阻塞等待 GPU 执行指令
